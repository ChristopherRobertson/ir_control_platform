# Salvage Matrix

This matrix is decisive by design. `KEEP` means preserve the asset or low-level knowledge with minimal structural change. `REWRITE` means preserve the knowledge but rebuild the implementation inside the new package boundary. `DELETE / DISCARD` means do not carry the structure or behavior into `ir_control_platform`.

## Explicit discard rules

| Legacy pattern or file | Classification | Why | Target destination in `ir_control_platform` | Rewrite notes |
|---|---|---|---|---|
| Old device-first navigation and dashboard structure | `DELETE / DISCARD` | Conflicts directly with the workflow-first product definition | none | Replace with Setup, Run, Hardware, Live Data, Analysis, Results, and Service. |
| Direct UI-to-device orchestration | `DELETE / DISCARD` | Violates the experiment-engine boundary | none | Coordinated behavior must move to `experiment-engine`. |
| UI-owned persistence and save-path authority | `DELETE / DISCARD` | Violates the data-pipeline boundary | none | Session and artifact writing belong to `data-pipeline`. |
| UI-owned processing and analysis | `DELETE / DISCARD` | Violates processing and analysis boundaries | none | Use persisted artifacts and headless jobs only. |
| Compatibility layers, fallback logic, and alternate execution paths | `DELETE / DISCARD` | The new system is single-path and fail-fast | none | Surface faults instead of routing around them. |
| Generic command passthrough endpoints | `DELETE / DISCARD` | They leak vendor-native APIs into the product surface | none | Replace with typed commands and queries. |
