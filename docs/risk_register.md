# Risk Register

This register captures the main technical and migration risks validated during Phase 1. All mitigations assume `Control_System` remains reference-only and `ir_control_platform` stays single-path and fail-fast.

| ID | Risk | Evidence | Impact if ignored | Required mitigation | Validation needed | Priority |
|---|---|---|---|---|---|---|
| `R1` | The target repo is still a docs-only skeleton | `ir_control_platform` has 19 non-`.git` files and no implementation files under the target packages | Phase 2 work can drift without shared contracts, scaffolding, or checks | Start Phase 2 with package scaffolding, typed contracts, and local validation commands before feature work | Confirm every target package has an entry point and check command | High |
| `R2` | Coordinated run behavior is split across UI, route, and controller layers | `ExperimentView.tsx`, `backend/src/modules/experiment/routes.py`, and HF2 recording logic all participate in run control | A port can accidentally recreate multiple run authorities | Put all coordinated sequencing into `experiment-engine` and make the UI a typed command/query client only | Simulator-backed start, stop, abort, and fault tests | High |
| `R3` | MIRcat sweep logic contains alternate execution paths and fallback behavior | `daylight_mircat/controller.py` uses advanced sweep, standard sweep, and optional software ping-pong fallback | Violates the single-path requirement and can hide firmware problems | Choose one supported sweep path per workflow and surface unsupported cases as faults | Bench validation for every supported sweep mode | High |
| `R4` | UI-owned settings and browser persistence currently hold authoritative state | `useMemoryState.ts`, `sessionStorage` in `ExperimentView.tsx` and `ScanModePanel.tsx`, plus `daylight_mircat/user_settings.json` | Recipe, device, and session truth will fragment across browser memory, route files, and hardware | Move configuration truth into contracts, presets, and session records outside the UI | Restart the UI during and after runs; verify state can be reconstructed from persisted artifacts | High |
| `R5` | HF2 recording still depends on live MIRcat state for axis data | `_get_current_wavenumber()` in `zurich_hf2li/controller.py` imports `mircat_controller` and reads current wavenumber | Replay and offline processing will be wrong or incomplete when MIRcat is absent | Make the run model and session manifest own axis metadata explicitly | Replay the same raw session without live hardware and compare results | High |
| `R6` | HF2 deployment assumptions are inconsistent | `zurich_hf2li/controller.py` header comment mentions API level 6, implementation sets `_api_level = 1`, and `agent_commands.json` reports `api_level: 1` | Wrong client or server assumptions can break connection and node access | Reconfirm actual LabOne server version, API level, host, and device ID before driver implementation | Bench connection test against the real instrument | High |
| `R7` | Raw LabOne node access leaks straight into the UI | `zurich_hf2li/routes.py` exposes `nodes/get` and `nodes/set`; `ZurichHF2LIView.tsx` hardcodes node maps | The UI can become a vendor-native console instead of a product surface | Replace raw node access with supported expert commands behind the driver boundary | Verify no `ui-shell` code constructs raw LabOne node paths | High |
| `R8` | Pico preview and autoset flow is heuristic and fallback-heavy | `picoscope_5244d/controller.py` uses autoset heuristics and immediate-capture fallback when trigger wait times out | Product behavior becomes non-deterministic and hard to test | Limit the supported acquisition modes and remove hidden fallback from the canonical path | Bench test supported preview modes and explicit no-trigger behavior | Medium |
| `R9` | Startup and autoconnect behavior is entangled with environment mutation | `start.bat` mutates `PATH`, performs setup checks, and legacy routes register startup auto-connect hooks | The new product can connect hardware without operator intent and behave differently per machine | Move startup knowledge into `platform` bootstrap and document prerequisites explicitly | Clean-machine bootstrap and startup review | High |
| `R10` | `hardware_configuration.toml` mixes concerns and contains bad values | The file mixes install paths, UI defaults, device limits, and unresolved placeholders; it also contains typos and at least one obviously wrong range (`tune_wavenumber_max = 20787.27`) | Copying it directly would bake invalid semantics into contracts and drivers | Normalize fields by ownership and revalidate them against manuals and bench facts | Field-by-field review against vendor docs and bench owners | High |
| `R11` | The legacy system has no authoritative session manifest or artifact graph | HF2 recording writes metadata comments and rows directly to files, but there is no session ID, artifact manifest, or export provenance | Replay, reprocessing, and exports will stay fragile and non-reproducible | Define `SessionManifest` and artifact records in Phase 2, then implement continuous session writing in Phase 3 | Reopen a partial or faulted session without the original UI state | High |
| `R12` | Bench-specific facts are still missing for several devices | `ACTION_REQUIRED.md` still requests COM ports, wiring, HF2 device ID, and SDK installation confirmations | Drivers may be structurally correct but unusable on real hardware | Treat missing facts as blockers, not as reasons to add code branches or guesses | Review and signoff with the hardware owners | High |
| `R13` | Historical raw-data layout assumptions may be too narrow | `convert_hf2li.py` assumes sequential sample/reference sections and filename-encoded wavenumber ranges | Later importers may silently misread historical datasets | Inventory every raw dataset family and reject unsupported layouts explicitly | Run the importer against all historical raw files | Medium |
| `R14` | Future device scope is broader than the code that actually exists | Manuals are present for Arduino and Continuum devices, but the repo has no meaningful implementations for them | Early milestones can get derailed by unbounded device scope | Keep those devices out of the first slice and treat them as later planning inputs only | Explicit milestone review before any additional driver work | Medium |

## Near-term gates before implementation expands

1. Freeze the first vertical slice to MIRcat plus HF2LI only.
2. Define contracts for recipes, readiness, run state, events, sessions, and artifacts before writing feature code.
3. Ban startup autoconnect, UI-owned persistence, raw node passthrough, and fallback branches from new code review.
4. Curate replay and golden-result fixtures from `Control_System/data`, `backend/data/raw`, `backend/data/converted`, and `tools/agent_commands.json`.
5. Capture the missing bench facts in controlled configuration, not in compatibility code.
